AI 开发酒店内部导游地图

从潮州古城有熊酒店项目，看技术选型与 AI 落地边界

按 → 键继续
判断 1

酒店导览不需要 App

一个 H5 网页 + 二维码就够了。

零下载
客人扫码即用，不用去应用商店，不用注册登录。
零审核
不需要小程序审核，改完推代码就生效。
零服务器成本
GitHub Pages 免费托管，一年费用为 0。

启发：低频使用的场景，别用 App。H5 的打开成本对客人最低，对酒店也最低。

判断 2

技术栈要压到最轻

不需要后端、不需要数据库、不需要地图 SDK。

前端
原生 HTML/CSS/JS，没有 React/Vue 框架，加载快，维护轻。
地图
自定义平面图 + CSS transform。高德百度没有酒店室内地图。
音频
预生成 MP3，不是实时 TTS。微信内置浏览器不支持实时语音合成。
数据
静态 JSON + 浏览器 localStorage。客人打卡只存在自己手机里。

启发：不要为"未来可能规模化"提前加复杂度。MVP 阶段，能少一个依赖就少一个。

判断 3

AI 有三件事很强，但不是万能

批量生成语音

30 个故事点音频，edge-tts 一键生成。改文案可重跑，省去录制档期。

结构化内容

把 Word、PPT、口述资料拆成统一字段：标题、摘要、故事、坐标、分类。

加速试错

从静态页面到语音、打卡、证书，一周迭代 3-4 版，降低决策风险。

边界：AI 不会自动知道哪把椅子、哪盏灯有故事。内容源头必须是人，AI 只是加速器。

建议

常熟有熊：先验证，再放大

5-8 个核心点位 → H5 MVP → 真实客人反馈
第一步
选大堂、庭院、餐厅、特色客房等 5-8 个点位，配平面图 + 语音 + 打卡。
第二步
在前台、电梯、客房放二维码，跑 2 周数据：多少人扫、听完几个点、愿不愿意分享。
第三步
验证通过再补全点位、换真人录音、上后端和多酒店品牌化。

启发：最便宜的验证方式成本几乎为零。确认客人真的愿意听后，再决定要不要规模化投入。
